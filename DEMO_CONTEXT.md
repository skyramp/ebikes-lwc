## What We Are Building

A demo that shows **Skyramp autonomously generating and executing functional tests
(API contract, integration, UI) when a change is made to a Salesforce application**,
catching breaking changes before they reach production.

The demo is targeting a May 5th RFP presentation for a large enterprise customer who
wants to see "AI-driven impact testing" on ERP systems (SAP, SFDC, Coupa).

## The Demo Application: E-Bikes

E-Bikes is an official Salesforce sample app simulating a fictional electric bicycle

- It has a real web UI (Experience Cloud portal for external resellers)

└── Experience Cloud Portal →  resellers browse catalog, place wholesale orders

Same database, two faces. A `Reseller_Order__c` record created by a reseller on the

## Salesforce Architecture — What You Need to Know

### The Platform
Salesforce is a hosted platform, not a self-hosted app. The application runs INSIDE
a Salesforce org (an isolated cloud environment). There is no Docker, no Kubernetes,
no traditional server. Deployment means pushing metadata into an org via the SF CLI.

### Key Concepts
- **Org:** The live running Salesforce environment. Has a permanent URL like
  `https://yourname-dev-ed.my.salesforce.com`
- **Metadata:** Everything in Salesforce is metadata — custom objects, fields,
  Apex classes, LWC components, permission sets. Stored as XML files in the repo.
- **SF CLI (`sf`):** The command-line tool for deploying to/from orgs.
  The old `sfdx` CLI is deprecated — always use `sf`.
- **Scratch Org:** Temporary org (expires in 30 days). Good for dev, bad for demos.
- **Developer Edition Org:** Free, permanent org. This is what we use for the demo
  so Skyramp has a stable URL to point at.

### The Two App Layers Relevant to Skyramp
1. **Apex REST endpoints** — custom backend code. Defined with `@RestResource`
   annotation. Hit at `/services/apexrest/your-endpoint/`
2. **SObject REST API** — auto-generated by Salesforce for every custom object.
   No code needed. Hit at `/services/data/v59.0/sobjects/ObjectName__c/`
   The `/describe` endpoint returns full schema — equivalent to an OpenAPI spec.

### Authentication
Every API call requires a Bearer token (OAuth). Two options:
- **Connected App (recommended):** OAuth client credentials flow.
  Create in Setup → App Manager. Give Skyramp `client_id` + `client_secret`.
- **Session token (quick demo):** Get via SF CLI, pass as secret. Expires.

## The Two Test Surfaces for Skyramp

### 1. REST API Surface
Base URL: `https://yourname-dev-ed.my.salesforce.com`

```
# Schema / contract endpoint (returns full field definitions)
GET /services/data/v59.0/sobjects/Reseller_Order__c/describe

# CRUD operations
GET    /services/data/v59.0/query?q=SELECT+Id,Name+FROM+Product__c
POST   /services/data/v59.0/sobjects/Reseller_Order__c/
PATCH  /services/data/v59.0/sobjects/Reseller_Order__c/{id}
DELETE /services/data/v59.0/sobjects/Reseller_Order__c/{id}

# Custom Apex REST (needs to be added to the repo — see below)
GET    /services/apexrest/products/
GET    /services/apexrest/orders/
```

**Note:** The standard SObject API exists automatically. Custom Apex REST endpoints
need to be written and added to the repo as part of the demo setup.

### 2. UI Surface
The **Experience Cloud portal** is the clean UI target. It is:
- A public-facing URL (can be accessed without Salesforce login for guest users)
- Standard HTML/CSS rendered in a browser
- Driveable by Playwright/Selenium like any web app

URL format: `https://yoursubdomain.my.site.com/EBikes`

The **internal Lightning App** is harder for UI testing:
- Behind Salesforce OAuth login
- Heavy Salesforce framework DOM
- Stick to Experience Cloud portal for UI tests in the demo

## What Needs to Be Added to the Repo for the Demo

The base E-Bikes repo does NOT have custom Apex REST endpoints. You need to add
them to make the API contract testing story compelling. Add these:

### 1. Custom Apex REST Controller for Products
File: `force-app/main/default/classes/ProductRestController.cls`
```apex
@RestResource(urlMapping='/products/*')
global class ProductRestController {
    @HttpGet
    global static List<Product__c> getProducts() {
        return [SELECT Id, Name, Description__c, MSRP__c, Category__c,
                Material__c, Level__c, Picture_URL__c
                FROM Product__c
                LIMIT 100];
    }
}
```
Accessible at: `GET /services/apexrest/products/`

### 2. Custom Apex REST Controller for Orders
File: `force-app/main/default/classes/OrderRestController.cls`
```apex
@RestResource(urlMapping='/orders/*')
global class OrderRestController {
    @HttpGet
    global static List<Reseller_Order__c> getOrders() {
        return [SELECT Id, Name, Status__c, Total_Amount__c,
                Reseller__c, CreatedDate
                FROM Reseller_Order__c
                LIMIT 100];
    }

    @HttpPost
    global static String createOrder(String resellerId, String status) {
        Reseller_Order__c order = new Reseller_Order__c(
            Reseller__c = resellerId,
            Status__c = status
        );
        insert order;
        return order.Id;
    }
}
```
Accessible at: `GET/POST /services/apexrest/orders/`

### 3. GitHub Actions Workflow for Skyramp
File: `.github/workflows/skyramp-ci.yml`
```yaml
name: Skyramp Impact Tests

on:
  pull_request:
    branches: [main]

jobs:
  deploy-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Salesforce CLI
        run: |
          npm install @salesforce/cli --location=global
          nodeInstallPath=$(npm config get prefix)
          echo "$nodeInstallPath/bin" >> $GITHUB_PATH

      - name: Authenticate to Dev Org
        run: |
          echo ${{ secrets.SF_AUTH_URL }} > ./SF_AUTH_URL.txt
          sf org login sfdx-url -f ./SF_AUTH_URL.txt -s -a targetorg

      - name: Deploy changes to Dev Org
        run: sf project deploy start -d force-app -o targetorg -w 10

      - name: Refresh sample data
        run: sf data tree import -p ./data/sample-data-plan.json -o targetorg

      - name: Publish Experience Cloud site
        run: sf community publish -n "E-Bikes" -o targetorg

      - name: Run Skyramp Tests
        run: |
          # Skyramp invocation goes here
          # skyramp test run \
          #   --api-target ${{ secrets.SF_ORG_URL }} \
          #   --ui-target ${{ secrets.SF_COMMUNITY_URL }} \
          #   --token ${{ secrets.SKYRAMP_API_KEY }}
```

## The Two Demo Scenarios

### Scenario A — Admin Change (the most resonant)
**Ticket:** "Add Shipping Priority field to all reseller orders for EU compliance"

**What happens:**
1. Admin adds `Shipping_Priority__c` (required picklist) to `Reseller_Order__c` in Setup
2. Developer runs `sf project retrieve start` — pulls field XML into codebase
3. PR opened — diff is just a few lines of XML, looks harmless
4. Skyramp reads diff, understands a required field was added to an order object
5. Skyramp runs:
   - POST to `/sobjects/Reseller_Order__c/` without the field → **FAILS** (400 error)
   - UI test on Experience Cloud order form → **FAILS** (field missing from form)
   - End-to-end order flow → **FAILS**
6. PR blocked. Developer knows exactly what to fix before production.

**Without Skyramp:** Merges to production Friday. Resellers can't place orders all
weekend. Support flooded Monday. 2-day incident traced back to a 3-line XML change.

### Scenario B — Developer Change
**Ticket:** "Add warranty information to product API for mobile app"

**What happens:**
1. Developer modifies `ProductRestController.cls` to add nested warranty object
2. PR opened — looks like a small additive change
3. Skyramp reads diff, detects API response shape changed
4. Skyramp runs:
   - Contract test against stored baseline → **WARNS** (response shape changed)
   - UI test on product catalog → **PASSES**
   - End-to-end order flow → **PASSES**
5. Warning surfaces: "downstream consumers may be affected by response shape change"

**Without Skyramp:** Additive change merges, silently breaks a strict-validation
integration that nobody remembered was consuming that endpoint.

## The Change Flows That Exist in Enterprise Salesforce

Understanding these is important for positioning Skyramp correctly:

| Flow | How change is made | Codebase updated? | Tested? |
|---|---|---|---|
| **Flow 1 (Admin-first)** | Admin in Setup → `sf retrieve` → PR | ✅ Yes | With Skyramp |
| **Flow 2 (Code-first)** | Developer writes code → PR | ✅ Yes | With Skyramp |
| **Flow 3 (Change Set)** | Admin manually promotes change set | ❌ Rarely | Never |
| **Flow 4 (Direct prod edit)** | Admin clicks directly in production | ❌ No | Never |
| **Flow 5 (Package install)** | ISV package installed from AppExchange | ❌ No | Never |
| **Flow 6 (Platform release)** | Salesforce auto-upgrades the platform 3x/year | N/A | Rarely |

**Skyramp's sweet spot for this demo:** Flows 1 and 2 — the PR is the universal
checkpoint regardless of whether the change came from an admin or a developer.

**The bigger commercial story:** Flows 3, 4, 5, 6 represent how most enterprise orgs
actually operate today. Getting them to Flow 1/2 discipline is step one; Skyramp is
the guardrail that makes that discipline worthwhile.

- Enable Dev Hub (Setup → Dev Hub)
- Enable Digital Experiences + choose subdomain (Setup → Digital Experiences → Settings)
